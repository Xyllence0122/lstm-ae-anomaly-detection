# 程式說明文件 — 半導體蝕刻製程異常偵測與串流早期預警

本文件詳細介紹整個專案的程式架構、使用框架、各模組的職責與執行方式。
對應的研究主題：**在邊緣端以因果式時間序列模型即時偵測製程行為模式異常，彌補終值監控的動態盲區**。V2 以 Sliding-Window LSTM-AE 為主模型，causal LSTM forecaster 為即時比較模型。

---

## 1. 專案概觀

### 1.1 這個程式在做什麼

本研究實作的 SPC X-bar baseline 只看每片 wafer 的**最終量測值**，因此「過程中有異常、但最後有達標」的 wafer 可能看不到（例如暫態到位過快、中段震盪、緩慢漂移）。這不代表所有 SPC 方法都只看終值。本專案：

1. 從真實 LAM 9600 金屬蝕刻機資料提取正常製程的統計特性
2. 用這些統計特性生成大量合成時間序列（含三種「SPC 看不到」的異常）
3. **只用合成正常資料**訓練 LSTM AutoEncoder，以重建誤差當異常分數
4. 與 SPC、Dense AE、Isolation Forest 同場比較
5. 回到**從未參與訓練的真實 wafer** 做最終驗證
6. 以 Sliding-Window LSTM-AE 逐筆重建最近視窗，執行 k-of-n 連續告警
7. 以 causal LSTM forecaster 比較快速暫態偵測能力
8. 以大量 normal、低 FPR、最終 holdout 與真實資料檢查部署可行性

核心邏輯：AE 只學過正常波形 → 遇到異常波形時重建不出來 → 重建誤差飆高 → 超過閾值就報警。

### 1.2 三層資料策略（防止資料洩漏的關鍵設計）

| 資料 | 用途 | 絕對不做的事 |
|---|---|---|
| 真實正常 wafer・統計組（60%，64 片） | Step 1 提取統計特性；Step 5 閾值校準 | 不當驗證負樣本 |
| 真實正常 wafer・保留組（40%，43 片） | Step 5 最終驗證的負樣本 | 不參與任何統計提取 |
| 真實 faulty wafer（20 片） | Step 5 最終驗證的正樣本 | 模型從頭到尾沒看過 |
| 合成資料（訓練 500 + 驗證 350 + 測試 500） | 模型訓練、模型選擇、合成基準測試 | 測試集每個 seed 只碰一次 |

### 3.1 `config.py` — 共用設定

- **路徑**：`DATA_MAT` 指向 LAM 9600 資料集的 .mat 檔（不在 repo 內）；`OUTPUT_DIR`、`FIGURE_DIR` 自動建立
- **`SELECTED_SENSORS`**：從 21 個工程變數中選 4 個監測 sensor，選擇邏輯是「設備控制迴路」觀點——被控變數（Pressure）、執行器（Vat Valve）、冷卻系統（He Press）、流量迴路（Cl2 Flow）——好處是解釋偵測結果不需要電漿化學知識
- **`PROCESS_STEPS = (4, 5)`**：只分析主蝕刻步驟（該資料集的慣例）
- **`MIN_WAFER_LEN = 60`**：以「原始長度」剔除紀錄不完整的 wafer（Step 1 與 Step 5 必須用同一規則，切分索引才對得上）
- **`set_plot_style()`**：統一所有圖表的字型（繁中）、配色、格線

### 3.2 `models.py` — 模型與工具函式（核心模組）

**四個模型類別：**

- `LSTMAutoEncoder`：Encoder LSTM 把整段序列壓成 16 維 latent 向量 → Decoder LSTM 從 latent 重建整段序列。重建長度跟隨輸入，**天然支援變長序列**（真實 wafer 長度 95~112 步不等）
- `DenseAutoEncoder`：把 (T, F) 攤平的全連接 AE。作為「沒有時序歸納偏置」的對照組，輸入須先重採樣成固定長度
- `LSTMForecaster`：在時間 `t` 只使用 `0..t` 樣本預測 `t+1`，可在線上持續保留 recurrent state
- `LSTMForecasterStep`：顯式輸入/輸出 hidden state 的包裝器，用於 TorchScript 邊緣推論

**工具函式（依呼叫順序）：**

| 函式 | 職責 |
|---|---|
| `train_collect_checkpoints()` | 只用正常資料訓練 AE，變長資料以「長度分桶」組 batch（LSTM 同一 batch 內長度須一致），每 20 epochs 存一份 CPU checkpoint |
| `pointwise_errors()` | 逐時間步、逐 sensor 的平方重建誤差 |
| `sensor_peak_scores()` | 對誤差沿時間軸做移動平均平滑後取**峰值**（局部異常不被整片平均稀釋），以 cumsum 實作 |
| `combine_peaks()` | 把 per-sensor 峰值合併成單一異常分數（max over sensors，可選校正） |
| `make_threshold()` | 兩種閾值規則：mean+3σ、p99 |
| `grid_select()` | 在驗證集掃「checkpoint × 平滑窗口 × 峰值校正 × 閾值規則」網格，以 F1 選最佳組合 |

**為什麼需要 checkpoint 網格而不是單純早停？** AE 訓練越久重建能力越強，強到連異常波形都能重建時偵測力反而下降；而且不同異常型態偏好不同收斂程度（漂移型偏好訓練久一點、震盪型偏好早一點），單一早停點兩頭不討好。所以保留整條訓練軌跡，用驗證異常集挑。

**裝置管理**：`DEVICE` 自動偵測 CUDA；所有 checkpoint 一律轉成 CPU tensor 存檔，訓練用 GPU、部署載入用 CPU 都不會出錯。

### 3.3 `01_sensor_stats.py` — 提取真實統計特性

讀 .mat → 篩掉過短 wafer（107 片正常）→ 以固定 seed（123）切 60/40 統計組/保留組 → 對統計組每個 sensor 提取：

- **mean profile**：正規化時間軸 101 點的平均波形（自然涵蓋 ramp 方向、step 4→5 轉換暫態）
- **within_wafer_std / lag1_autocorr**：扣掉 profile 與片間 offset 後的殘差統計（噪聲強度與自相關）
- **between_wafer_std**：片間 offset 的變異（每片 wafer 的基準值略有不同）
- **quant_step**：量化階距（真實訊號是整數量化的，這點對縮小 synthetic→real gap 很重要）
- **transient_amp**：製程前段暫態幅度（決定「暫態到位過快」異常適用於哪些 sensor）
- **sampling**：由 `Time` 欄實測的取樣間隔分布；目前中位數 1.0200，若單位為秒約 0.980 Hz

輸出 `sensor_stats.json` + 真實波形圖。**切分索引也存進 json**，Step 5 直接沿用，保證兩步看到同一份切分。

### 3.4 `02_generate_synthetic.py` — 合成資料生成器（v2.1）

每片合成 wafer = **真實 mean profile 骨架**（時間軸縮放到隨機長度 95~112）+ 片間 offset + **平穩 AR(1) 殘差噪聲** + **整數量化**。

三種異常（每片隨機影響 1~2 個 sensor，**最終值都在正常範圍** → SPC 看不到）：

| 類型 | 做法 | 對應真實情境 |
|---|---|---|
| A 暫態到位過快 | 暫態段時間軸壓縮 2.5~4 倍 + 過衝 ringing（只發生在真實有暫態的 sensor：Pressure、Vat Valve） | 控制迴路響應過快；不等同於已驗證的熱損傷 |
| B 過程震盪 | 中段 2.5~4σ 阻尼震盪，結束前恢復 | 迴路不穩定、暫時性干擾 |
| C 緩慢漂移 | 線性漂移，最終偏移 2.0~2.8σ（管制界限內） | 元件老化、慢速失準——SPC 最難抓的 |

資料集切分：訓練 500 正常 / 驗證 200 正常 + 各 50 異常（**模型選擇專用**）/ 測試 200 正常 + 各 100 異常（**只在最終評估碰一次**）。同時輸出變長版（LSTM 用）與重採樣固定長度版（Dense AE / IF 用）。

### 3.5 `03_train_lstm_ae.py` — 訓練（5 個隨機種子）

對每個 seed（42~46）：先設定 seed 再建立模型，訓練 400 epochs → `grid_select()` 選最佳組合 → 在測試集評估**一次** → 結果與權重單獨存檔。快取綁定資料 SHA-256、seed 與超參數，只有簽章一致才會跳過。

最後彙總 5 seeds 的 mean ± std（單一 seed 的數字不可信），並把**驗證 F1 最高的 seed** 打包成發佈模型 `lstm_ae.pt`（含標準化參數 mu/sd、平滑窗口、校正向量、閾值——部署所需的一切）。

標準化：z-score，**只用訓練集統計**（測試集資訊不可滲入）。

### 3.6 `04_compare_methods.py` — 四方法比較

| 方法 | 輸入 | 閾值選擇 |
|---|---|---|
| SPC X-bar | 每片 wafer 的**最終值** | ±3σ 管制界限（傳統做法，無需選擇） |
| Dense AE | 固定長度重採樣版 | 與 LSTM-AE **完全相同**的 grid_select 協議 |
| Isolation Forest | 固定長度攤平向量 | 閾值規則同樣以**驗證異常 F1** 挑選（協議一致，對比才公平） |
| LSTM-AE | 變長原始序列 | 直接載入 Step 3 發佈模型的設定 |

輸出指標表（CSV/JSON）、四方法長條圖，以及論文最重要的**SPC 盲點示範圖**：挑一片「LSTM-AE 抓到、SPC 漏掉」的緩慢漂移 wafer，三聯圖呈現 (a) 波形上漂移量很小 (b) SPC 管制圖上最終值完全在界限內 (c) LSTM-AE 重建誤差在漂移段明顯超過閾值。

### 3.7 `05_validate_real_data.py` — 真實資料最終驗證

模型從沒看過真實資料，這一步回答「合成資料訓練的模型在真實世界到底行不行」。評估兩種情境：

1. **直接遷移**：閾值直接沿用合成資料的設定 → 考驗 synthetic→real domain gap 有多大
2. **重新校準**：用統計組真實 wafer 重算校正與閾值 → 實務部署會做的事

負樣本 = 43 片保留組真實正常 wafer；正樣本 = 20 片真實 faulty wafer。同時報告 SPC、Dense AE 與 causal forecaster 在同一份真實資料上的表現、AUC，以及「故障發生在有監測/未監測 sensor」的分組偵測率（量化 sensor 覆蓋的取捨）。Forecaster 額外報告成功偵測樣本的首次告警製程進度；真實資料沒有 fault-onset timestamp，因此不計真實 detection delay。

程式開頭有一致性檢查：如果真實 wafer 數量與 Step 1 記錄不符（切分索引會錯位），直接報錯要求重跑 Step 1。

### 3.8 `06_sampling_rate_sweep.py` — 取樣率掃描實驗（回答「微小快速異常看不看得到」）

**動機（白話版）**：sensor 就像一台固定快門速度的監視器。`Time` 欄實測中位取樣率約 0.980 Hz（時間單位假設為秒），所以秒以下事件的細節沒有被記錄。這不是模型能力問題，而是可觀測性的物理限制。直接生成「毫秒級合成資料」宣稱能偵測真實毫秒異常沒有驗證依據；本實驗改問：**在明確的連續時間情境模型下，同一個異常取樣多快才抓得到？**

**做法**：

- 建一個**連續時間情境模型**：一階設定點響應 + OU（Ornstein-Uhlenbeck）噪聲。穩態位準、噪聲強度 σ、片間變異、量化階距來自真實統計；噪聲相關時間 θ 由實測中位間隔與 lag-1 自相關反推。正常 2 秒到位是顯式情境假設，不是 LAM 9600 的溫度實驗。
- 異常 = 到位時間壓縮成 1 / 0.5 / 0.2 / 0.05 秒 + 與事件時間尺度綁定的過衝 ringing
- 對每個取樣率（0.5 / 1 / 2 / 5 / 10 / 20 Hz）**各訓練一個 LSTM-AE**（協議與主管線相同），再對每種異常速度評估偵測率
- 每個取樣率的結果單獨存檔（`outputs/sweep/`），中斷可續跑

**產出與論點**：`06_waveform_demo.png` 呈現同一異常在不同取樣率下的可觀測差異；`06_detection_vs_rate.png` 顯示此情境模型中的偵測率與取樣率關係。對 0.01 秒事件的數百 Hz 推估只能標為情境外插。是否需要邊緣處理仍應以實際 sensor 數、資料格式、網路頻寬與延遲預算量化，不能只用一句「無法上雲」取代工程計算。

### 3.9 `07_streaming_early_warning.py` — 因果式串流早期預警

`LSTMForecaster` 只用正常合成序列學習一步預測。每筆新樣本到達後，系統計算逐 sensor 預測誤差，以 causal trailing window 平滑並取 sensor 最大值。異常 wafer 只有在合成注入 onset 之後首次越界才算命中；onset 之前越界另列為 pre-onset alarm。

腳本會執行 5 seeds、保存資料/設定簽章快取、選出驗證 F1 最高的 release seed，並輸出：

- `streaming_lstm_forecaster.pt`：完整 checkpoint、標準化與閾值設定
- `streaming_lstm_step.ts`：顯式 recurrent state 的 TorchScript step model
- `streaming_early_warning.json`：各 seed、各異常召回、首次告警與 CPU 微基準
- `07_streaming_early_warning.png`：release model 的事件召回與告警時間線

本機微基準只計一次 recurrent step，不含 sensor I/O、前處理、平滑、警報傳輸，也不能替代 Raspberry Pi 實測。

### 3.10 `08_train_sliding_window_lstm_ae.py` — V2 主模型

載入 Step 3 只以正常完整製程軌跡訓練的 5-seed LSTM-AE，在時間 `t` 僅輸入最近的因果視窗。驗證集比較 `W=8/16/32/64`、重建誤差彙整、變化量誤差與 `1/1、2/3、3/5` 連續告警。發布設定為 seed 44、W=32、mean reconstruction error、3-of-5。

輸出 `sliding_window_lstm_ae.pt/.ts`、`sliding_window_metrics.json` 與 `08_sliding_window_results.png`。腳本同時驗證 eager Python 與 TorchScript 逐筆 runtime 的分數及告警一致性。

### 3.11 `09_compare_online_models.py` — 低誤報與盛行率

另產生 5,000 片 normal 校準及 10,000 片 normal 測試資料，評估 1%、0.5%、0.1% 目標 FPR，並投影異常率為 5%、1%、0.1% 時的警報 Precision。輸出 `prevalence_evaluation.json` 與 `09_low_fpr_comparison.png`。

### 3.12 `10_evaluate_locked_holdout.py` — 最終封存測試

模型凍結後，以不同亂數種子產生 10,000 normal 與 A/B/C 各 1,000 的最終 holdout，報告 Wilson 95% CI 並保存模型 SHA-256。若報告已存在，腳本只重畫圖，不重新產生結果。

### 3.13 `edge_window_runtime.py` — 邊緣逐筆推論

`SlidingWindowAnomalyDetector.update()` 會驗證 shape、有限值及遞增時間戳，維護最近 W 點、執行 TorchScript、計算逐 sensor 分數與 k-of-n 告警。每片 wafer、recipe 或監控區段切換時必須呼叫 `reset()`。

## 4. 如何執行

### 4.1 準備

1. 建立虛擬環境並安裝：`.\.venv\Scripts\python.exe -m pip install -r requirements.txt`
2. 以 `LAM9600_DATA_MAT` 環境變數指定 `MACHINE_Data.mat`；未設定時沿用 `config.py` 的相容預設

### 4.2 依序執行

```bash
python 01_sensor_stats.py       # 幾秒鐘
python 02_generate_synthetic.py # 幾秒鐘
python 03_train_lstm_ae.py      # CPU 約 15~25 分鐘（5 seeds）；有 CUDA GPU 會快很多；可中斷續跑
python 04_compare_methods.py    # 幾分鐘（要訓練 Dense AE）
python 05_validate_real_data.py # 幾秒鐘
python 06_sampling_rate_sweep.py # 獨立實驗；CPU 約 30~60 分鐘，可中斷續跑
python 07_streaming_early_warning.py # causal forecaster + TorchScript + CPU benchmark
python 08_train_sliding_window_lstm_ae.py # V2 主模型 + edge artifact
python 09_compare_online_models.py # 大量 normal + low-FPR/prevalence
python 10_evaluate_locked_holdout.py # 凍結模型最終 holdout
```

`06` 只依賴 `sensor_stats.json`，跑完 `01` 之後任何時候都能執行，不必等 `03~05`。

### 4.3 重跑注意事項

- `03` 與 `07` 的 seed 快取包含資料 SHA-256 與主要設定簽章；不一致時會自動重訓，不必手動搬移舊快取

- 只想重跑某個 seed：刪掉 `outputs/seeds/seed_XX.json` 和 `.pt` 即可

## 6. 方法論設計重點（口試常問）

- **為什麼只用正常資料訓練？** 異常樣本稀少且型態未知，正常資料建模符合 one-class anomaly detection 情境；但本專案用 labeled synthetic anomaly validation set 選 checkpoint 與閾值規則，所以完整流程是 semi-supervised，不是完全無監督
- **為什麼用「平滑誤差峰值」而不是整片平均誤差？** 局部異常（如中段震盪）只占整段序列一小部分，取平均會被稀釋；先平滑（抗單點噪聲）再取峰值最敏感
- **為什麼測試集只碰一次？** 網格選擇用的是獨立的驗證異常集，測試集若參與選擇就是洩漏，數字會虛高
- **為什麼報告 5 seeds 的 mean ± std？** 小模型 + 小資料的訓練結果對初始化敏感，單一 seed 的好成績可能是運氣
- **合成資料的正當性？** 正常骨架與噪聲統計來自真實資料，異常則是可控情境假設。真實 holdout 只能檢查 synthetic-to-real domain gap，不能把合成 A/B/C 宣稱成已被真實故障驗證
- **能偵測毫秒級的異常嗎？** 要區分**推論延遲**與**可偵測時間尺度**。Forecaster 在本機的一步推論是次毫秒微基準，但原始資料中位取樣率僅約 0.980 Hz，沒有秒以下證據。本專案不宣稱已偵測真實毫秒事件；`06` 只是在顯式情境模型下研究取樣率敏感度
- **為什麼不直接生成毫秒級合成資料來訓練？** 生成毫秒資料的依據只有 1 Hz 的真實統計，秒以下的物理特性等於用猜的，而且沒有任何真實毫秒資料可以驗證猜得對不對——等於自己畫慢動作影片再宣稱 AI 在裡面抓到東西。`06` 的設計把「猜的部分」縮到最小並顯式陳述為假設，宣稱的結論（取樣率與可偵測性的關係）不依賴那些假設的細節

# 研究模型與論文主張檢核

## 信心程度

- 程式與實驗流程檢核：高。依據為實際程式、輸出檔、原始 `.mat` 結構與自動測試。
- LAM 9600 製程物理意義：中。資料欄位與故障名稱可核對，但沒有設備 recipe、腔體設計及專家標註可供完整因果解釋。
- 「升溫速率造成晶圓熱應力損傷」：未知。本專案資料沒有溫度 sensor 或熱應力/良率標籤，必須另以溫度資料與文獻驗證。
- Raspberry Pi 即時性：未知。目前只有本機 CPU 微基準，尚未在 Raspberry Pi 實測。

## 建議的論文主軸

建議將核心貢獻寫成：

> 在邊緣端以因果式時間序列模型即時辨識製程暫態行為異常，補足終值監控無法分辨「同樣達標、過程不同」的動態盲區。

SPC、合成資料與 LSTM-AE 應作為支撐此主軸的實驗設計，不宜各自宣稱為並列創新。

## 現有模型是否搭配正確

| 方法 | 現有角色 | 能否途中告警 | 檢核結論 |
|---|---|---:|---|
| SPC X-bar（終值） | 動機與盲區示例 | 否 | 合理，但只代表本研究設定的終值監控，不代表所有 SPC 方法 |
| Isolation Forest | 非神經、固定長度基準 | 否 | 可保留；現有攤平輸入會弱化局部時間結構，需誠實說明 |
| Dense AE | 無 recurrent bias 的 AE 基準 | 否 | 必須保留；它可檢驗成效究竟來自整段曲線，還是 LSTM 本身 |
| LSTM-AE | 完整序列重建與離線確認 | 否 | 適合 wafer 結束後評分，不能直接支持「即時預警」 |
| Causal LSTM forecaster | 逐點預測與線上告警 | 是 | 與邊緣提前預警目標最一致，應作為即時模型主角 |

最重要的判斷是：LSTM-AE 並非因名稱含有 LSTM 就天然具備串流能力。原架構以完整序列編碼後再重建，告警分數是整片最大重建誤差，因此必須等序列結束。新增的 causal forecaster 在時間 `t` 只使用 `0..t` 樣本預測 `t+1`，才符合線上決策的資訊限制。

## 已核實的資料事實

- 原始 LAM 9600 `.mat` 內有 108 片 calibration normal 與 21 片 test faulty。
- 套用專案 `MIN_WAFER_LEN=60` 後，可用資料為 107 片 normal 與 20 片 faulty。
- 107 片 normal 固定切成 64 片統計/校準組與 43 片最終保留組。
- 統計組 step 4+5 的序列長度為 95 到 112 點，平均約 100.7 點。
- `Time` 欄正間隔中位數為 1.0200，P05 到 P95 為 0.9961 到 1.0488；若該欄單位為秒，中位取樣率約 0.980 Hz。
- 目前模型使用 Cl2 Flow、He Press、Pressure、Vat Valve；沒有 Temperature 欄位。

因此，「0.001 秒升到 200°C」不能用目前資料直接驗證。專案能驗證的是更一般化的命題：最終設定點相同時，暫態到位速度、震盪與漂移是否仍可由時間序列模型辨識。

## 已修正的方法學問題

1. **異常命名不精確**：Type A 已由「升溫過快」改為「暫態到位過快」，避免把 Pressure 暫態誤寫成溫度實驗。
2. **未保存異常起點**：合成資料現在保存 onset、end 與 affected sensor，才能計算真正的首次告警延遲。
3. **注入前誤報被算成命中**：目前只有 onset 之後的越界才算 true positive，pre-onset alarm 另行報告。
4. **seed 未控制初始權重**：原程式在模型建立後才設定 seed；現在改為先設 seed 再建立模型。
5. **舊快取可能被誤用**：模型快取現在綁定資料 SHA-256、seed 與超參數；不相符時自動重訓。
6. **取樣率只靠假設**：現在由 `Time` 欄計算實測間隔，取樣率掃描使用實測中位間隔反推 OU 相關時間。
7. **無 GUI 環境無法繪圖**：所有實驗改用 headless Matplotlib backend。

## 串流模型正式結果

合成測試集固定為 200 片 normal，加上 Type A/B/C 各 100 片；模型只用 normal 訓練，但 checkpoint、平滑窗口及閾值規則會使用獨立的 labeled synthetic anomaly validation set 選擇。因此模型訓練是 unsupervised，完整模型選擇流程應稱為 semi-supervised，不應只寫「完全無監督」。

5 seeds 的 causal forecaster 結果：

| 指標 | Mean | Std |
|---|---:|---:|
| Precision | 0.971 | 0.007 |
| Recall | 0.663 | 0.033 |
| F1 | 0.787 | 0.022 |
| Normal wafer FPR | 0.030 | 0.008 |
| Pre-onset alarm rate | 0.000 | 0.000 |

| 異常 | Recall mean | 首次告警進度中位數的跨 seed 平均 | 告警時剩餘序列比例 |
|---|---:|---:|---:|
| A：暫態到位過快 | 0.720 | 0.087 | 0.913 |
| B：過程震盪 | 0.576 | 0.557 | 0.443 |
| C：緩慢漂移 | 0.692 | 0.837 | 0.163 |

「剩餘序列比例」只表示離製程紀錄結束還有多少，不等同於距離晶圓損傷或事故還有多少時間。後者需要真實 damage/yield event timestamp 才能定義。

## 完整序列模型比較

完整序列 LSTM-AE 的 5-seed 合成測試平均為 Precision `0.976 +/- 0.008`、Recall `0.649 +/- 0.025`、F1 `0.780 +/- 0.020`、FPR `0.024 +/- 0.007`。使用發佈 seed 43 與其他基準比較：

| 方法 | Precision | Recall | F1 | FPR |
|---|---:|---:|---:|---:|
| Final-value SPC X-bar | 0.882 | 0.050 | 0.095 | 0.010 |
| Dense AE | 0.932 | 0.773 | 0.845 | 0.085 |
| Isolation Forest | 0.667 | 0.040 | 0.075 | 0.030 |
| LSTM-AE | 0.986 | 0.697 | 0.816 | 0.015 |

Dense AE 在這次單一 release-seed 比較中的 F1 較高，代價是正常 wafer FPR 由 LSTM-AE 的 0.015 增加到 0.085。因此可討論靈敏度與誤報成本的取捨，不能宣稱 recurrent 架構必然勝出。

## 真實資料檢查

真實保留組包含 43 片 normal 與 20 片 faulty：

| 方法/設定 | FPR | Recall | ROC-AUC |
|---|---:|---:|---:|
| LSTM-AE direct transfer | 0.140 | 0.800 | - |
| LSTM-AE real-normal recalibration | 0.047 | 0.700 | 0.907 |
| Final-value SPC X-bar | 0.000 | 0.250 | - |
| Dense AE | 0.000 | 0.650 | 0.923 |
| Causal LSTM forecaster | 0.047 | 0.650 | 0.910 |

Forecaster 在成功偵測樣本中的首次告警製程進度中位數為 0.099，但資料沒有經核實的 fault onset；這個數字只能描述「在紀錄中的何處告警」，不能稱為真實故障提前量。LSTM-AE 重新校準後，監測 sensor 相關 fault recall 為 1.0、未監測 sensor fault recall 為 0.5，顯示 sensor coverage 會直接限制系統能力；由於樣本數很小，這兩個比例不宜外推成產線效能。

## 邊緣效能可以與不可以宣稱的內容

目前 release model 有 18,180 個參數，TorchScript step artifact 約 78.4 KiB。本機 Windows CPU、單執行緒、2,000 次 recurrent step 微基準為：

- mean：158.1 microseconds
- p50：138.4 microseconds
- p95：237.4 microseconds

這只包含一次 recurrent inference step，不包含 sensor I/O、標準化、平滑、警報傳輸、排程延遲或作業系統負載。論文可以將它列為「開發主機微基準」，不能寫成 Raspberry Pi 延遲或完整端到端毫秒告警。

## 論文中應避免的主張

- 不要寫「已證明可避免良率損失或重大事故」。目前沒有 yield、damage 或事故結果標籤，只能寫「目標是降低風險，尚待產線驗證」。
- 不要寫「已驗證 0.001 秒升溫」。原始資料約 1 Hz，毫秒事件根本未被量測。
- 不要寫「LSTM-AE 全面優於 Dense AE」。Dense AE 是必要且有競爭力的基準，結果應以實際數字呈現。
- 不要把終值 SPC 當成所有傳統方法。應明確限定為本研究實作的 final-value X-bar baseline。
- 不要將合成異常當成真實物理故障。合成資料只驗證演算法在明確假設下的行為。

## 下一階段實驗

1. 在 Raspberry Pi 上量測 p50/p95/p99、CPU、RSS memory、功耗、持續運行掉樣率與端到端告警延遲。
2. 取得含溫度與 setpoint 的高頻資料，至少保留 command、measured value、取樣時間與 fault/yield label。
3. 對真實保留組使用 bootstrap confidence interval；20 片 faulty 的單一比例不應過度解讀。
4. 比較 causal forecaster、derivative/EWMA/CUSUM 與 temporal convolution/Transformer；所有方法使用相同切分與閾值協議。
5. 以 run-to-failure 或工程師確認的 fault onset 定義 detection delay 與 actionable lead time。

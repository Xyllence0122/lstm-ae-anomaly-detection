# V4 即時邊緣異常偵測系統實作說明

## 1. 研究目標

論文題目為「基於邊緣運算與時間序列模型之半導體製程即時異常偵測系統
之架構」。V4 的可驗證目標是：

1. sensor 資料逐筆到達時，以因果方式推論，不讀取未來資料。
2. 在邊緣端執行 V3.2 的多尺度 Sliding-Window LSTM-AE。
3. 對 schema、時間、缺值及 stream 邊界採取 fail-closed 契約。
4. 警報發生時保存波形、製程識別及完整模型 provenance。
5. 提供能在 Raspberry Pi 5 上直接執行的量測工具。

V4 是論文用 edge runtime prototype，不是量產設備的安全連鎖系統。

## 2. 模型版本策略

V4 不修改下列 V3.2 鎖定內容：

- 模型權重。
- 9個因果特徵的定義。
- 3組多尺度 profile。
- ensemble threshold `1.1795640023513194`。
- V3.2 locked holdout及歷史真實資料稽核報告。

因此 V4 不會產生新的模型 Accuracy、Recall 或 F1。論文中的模型效果仍引用
`outputs/v3/final_holdout_v3_2.json`：

| 指標 | V3.2 鎖定結果 |
|---|---:|
| FPR | 0.06% |
| Precision | 99.71% |
| Recall | 68.23% |
| F1 | 81.02% |
| Accuracy | 92.62% |
| AUC | 96.47% |
| A/B/C Recall | 48.1% / 74.2% / 82.4% |

這些數字來自同一生成器家族的合成 holdout，不代表量產現場表現。

## 3. 即時資料流程

```text
具名 sensor sample + timestamp + wafer/recipe/equipment ID
  -> schema、有限值、時間單調及 cadence 檢查
  -> raw + per-sample difference + elapsed sample phase
  -> feature normalization
  -> W8 與 W64 trailing windows
  -> W8-max、W8-mean、W64-mean reconstruction errors
  -> per-feature calibration
  -> profile persistence
  -> maximum normalized profile score
  -> strict score > threshold
  -> alarm result + top evidence features
  -> JSONL 警報前後波形保存
```

同一個8點視窗只執行一次模型，再供max與mean profile共用，避免重複推論。
任何結果都包含模型版本、TorchScript hash、manifest hash及schema hash。

## 4. 時間與取樣契約

來源 LAM 資料的取樣間隔統計為：

- median：約 `1.0199` 秒。
- p05：約 `0.99414` 秒。
- p95：約 `1.0488` 秒。

V3.2 特徵是 `x[t] - x[t-1]`，不是除以實際 `delta-t` 的物理變化率。
因此 V4 不任意改變已鎖定模型，而是在 runtime 強制固定 cadence：

- 每筆都必須提供有限數值時間戳。
- 時間戳必須嚴格遞增。
- 相鄰間隔必須在manifest設定範圍內。
- 超出範圍的資料拒絕推論，不會靜默產生可能錯誤的分數。
- 提供sensor timeout health check。

這個契約可支援約1 Hz情境下的動態異常研究，但不能證明0.1秒事件可被觀測。
未來若取得高頻且有jitter的真實資料，必須改成固定頻率重採樣或
timestamp-aware rate，並建立新模型版本、normal calibration及新holdout。

## 5. Stream 與事件契約

每個 wafer、recipe、機台或監控製程區段開始前，必須呼叫：

```python
detector.start_stream(wafer_id, recipe_id, equipment_id)
```

這會清除前一個 stream 的視窗、差分基準、persistence及時間狀態，避免不同
wafer資料串接。缺少任何識別欄位時拒絕啟動。

`JsonlAlarmRecorder` 僅在alarm由false轉為true時建立新事件，預設保存：

- 警報前64筆加上警報當下樣本。
- 警報後16筆。
- wafer、recipe及equipment ID。
- 分數、threshold、觸發profile及前三個異常證據特徵。
- 模型、manifest及sensor schema SHA-256。

程式關閉時尚未收滿後續樣本的事件會標記為
`truncated_on_shutdown`，不會假造缺少的資料。

## 6. 部署產物與一致性

執行：

```powershell
.\.venv\Scripts\python.exe 26_build_v4_deployment.py
```

會建立：

- `outputs/v4/sliding_window_lstm_ae_v4.ts`
- `outputs/v4/runtime_parity_v4.json`
- `outputs/v4/deployment_manifest_v4.json`
- `outputs/v4/deployment_manifest_v4.sha256`

manifest 綁定V3.2 checkpoint、TorchScript、統計量、normal calibration、
locked report、parity report及相關程式碼。runtime載入時會驗證所有hash。

parity資料只來自selection cohort，不會用locked holdout調整任何參數。
報告必須同時滿足：

- eager與TorchScript輸出最大差異不超過 `1e-5`。
- 離線與逐筆profile score最大差異不超過 `1e-5`。
- alarm decision mismatch為0。
- 每條序列的首次告警sample完全一致。

## 7. Raspberry Pi 5驗證

在Pi 5上執行：

```bash
python 27_benchmark_v4_runtime.py \
  --output outputs/v4/pi5_benchmark_v4.json \
  --sequences 1000 \
  --torch-threads 1
```

只有報告中的 `hardware.is_raspberry_pi_5=true` 才能在論文稱為Pi 5結果。
開發機產生的報告只能用於程式除錯。Pi 5論文實驗至少應報告：

- model load時間。
- update及實際推論的p50、p95、p99、maximum latency。
- 每秒處理樣本數。
- RSS memory。
- CPU溫度。
- 連續24至72小時的例外、掉樣與重啟次數。
- 網路斷線時事件是否持續寫入本地端。

功耗需要外接電表或可驗證的Pi感測方式，目前工具不會猜測功耗。

## 8. 尚未完成及不得宣稱的項目

1. 尚未取得全新的獨立真實normal/faulty資料。
2. 尚未在實體Pi 5量測延遲、溫度、功耗及長時間穩定性。
3. CSV replay已完成，但MQTT、OPC UA或機台專用協定仍取決於實際設備。
4. V3.2 Recall為68.23%，Type A Recall為48.1%，仍不能宣稱量產可用。
5. 約1 Hz資料不能支持次秒級故障偵測主張。
6. V4沒有自動線上學習；未經審核的異常資料不得直接更新正式模型。

雲端持續優化若列入未來工作，應採取資料版本化、工程師標註、離線重訓、
新holdout驗證、shadow deployment及可回滾model registry，而不是讓Pi直接
以未確認資料自行更新權重。
